# @extropy/node-handshake

**Sandbox-to-sandbox node handshake.** Proof-of-concept for the v3.1 substrate.

This package implements the minimum viable communication layer between two Extropy Engine nodes — for example, a VPS-hosted node and a local laptop node. It exists to prove the protocol shape end-to-end and to give Randall a working harness for testing today, while the full libp2p / native gossip layer is being designed.

**Status:** Skeleton — wire protocol defined; reference implementation in progress.

See [`docs/VPS_NODE.md`](../../docs/VPS_NODE.md) for the deployment posture this is built against.

## What it does

1. **Discovery.** Two nodes find each other via shared rendezvous URL or static peer list. (No DHT yet.)
2. **Handshake.** Each node presents:
   - Its node ID (Ed25519 public key)
   - Its version and supported protocol features
   - Its declared role (`vps`, `local`, `validator`, `observer`)
   - A signed timestamp (replay defense)
   - Optional: DID + ZKP if identity layer is online
3. **Capability exchange.** Both nodes advertise:
   - Which packages they run
   - Which entropy domains they can validate
   - Their current load
   - Whether they accept inbound claims/quests
4. **Channel establishment.** A signed session channel is opened over HTTPS (production: libp2p + Noise; sandbox: HTTPS + signed bodies).
5. **DAG replay.** Either node can request DAG segments by `(start_index, end_index)` and verify each vertex signature.
6. **Liveness.** Periodic signed heartbeats; missed heartbeats trigger session re-establishment.

## What it does NOT do (yet)

- No DHT discovery (`validation-neighborhoods` will eventually grow into this)
- No multi-hop routing
- No NAT traversal beyond what HTTPS allows
- No production-grade peer scoring
- No automatic peer banning

These are tracked in [`docs/GAPS.md`](../../docs/GAPS.md) under "P2P substrate."

## Wire protocol (sandbox)

All messages JSON-over-HTTPS, signed with the sender's Ed25519 key. Signatures are over the canonicalized JSON body.

### `POST /hello`

Request:
```json
{
  "nodeId": "ed25519:<base64-pubkey>",
  "version": "0.1.0",
  "spec": "v3.1",
  "role": "vps" | "local" | "validator" | "observer",
  "features": ["dag-replay", "claim-relay", "heartbeat"],
  "ts": "2026-05-03T18:00:00Z",
  "nonce": "<uuid>",
  "signature": "<base64>"
}
```

Response:
```json
{
  "ok": true,
  "peerNodeId": "ed25519:<base64-pubkey>",
  "version": "0.1.0",
  "spec": "v3.1",
  "role": "vps",
  "features": ["dag-replay", "claim-relay", "heartbeat", "epistemology-witness"],
  "sessionId": "<uuid>",
  "ts": "2026-05-03T18:00:00Z",
  "signature": "<base64>"
}
```

### `POST /capabilities`

Mutual exchange of packages, supported domains, current load.

### `POST /dag/replay`

```json
{
  "sessionId": "<uuid>",
  "fromIndex": 0,
  "toIndex": 100,
  "ts": "2026-05-03T18:01:00Z",
  "signature": "<base64>"
}
```

Response: streamed DAG vertices, each individually signed by their original author and verifiable independent of the responder.

### `POST /heartbeat`

```json
{
  "sessionId": "<uuid>",
  "ts": "2026-05-03T18:02:00Z",
  "depth": 12345,
  "signature": "<base64>"
}
```

## Running locally + against the VPS

```bash
# On VPS (or sandbox VPS):
pnpm --filter @extropy/node-handshake start
# default: PORT=4200, NODE_ROLE=vps, NODE_PRIVATE_KEY in env

# On local laptop:
NODE_ROLE=local PEER_URL=https://your-vps-host:4200 \
  pnpm --filter @extropy/node-handshake client:hello
```

Outputs the full handshake exchange and verifies signatures from the peer.

## Security notes for the sandbox

- The sandbox protocol is HTTPS + body signing. **It is not a hardened production transport.** Production will move to libp2p + Noise framing.
- The Ed25519 keys generated by the sandbox are fine for testing. Real node keys should be derived from the participant's DID material once Identity is online.
- Do NOT expose this service on a public IP without HTTPS termination and a rate limiter.

## Why this exists

Randall's near-term goal is to confirm protocol shape against actual two-node reality before scaling out. This package is the smallest plausible thing that exercises:

- Cryptographic identity at the node level
- Mutual capability exchange
- DAG portability between nodes
- Heartbeat-driven liveness

If this works between the VPS and the laptop, the protocol shape is real. If it doesn't, we find out before scaling further.
